## v2.0.0 → v2.0.2 完整变更记录

### 1. Supabase 模块全量移除

**背景：** 邮件验证码登录是纯本地 OTP 验证 + MailerSend/Brevo 发送，不依赖 Supabase。旧代码残留大量 Supabase SDK 引用（`SupabaseConfig`、`SupabaseRestService`、`SupabaseOtpService`、`HybridOtpService`、`WorkerOtpService`、`PostgreSQLAuthService`、`DefaultDatabaseConfig`）。

**删除的文件（9 个）：**
- `UEModManager/Services/SupabaseConfig.cs`
- `UEModManager/Services/SupabaseRestService.cs`
- `UEModManager/Services/SupabaseOtpService.cs`
- `UEModManager/Services/HybridOtpService.cs`
- `UEModManager/Services/WorkerOtpService.cs`
- `UEModManager/Services/PostgreSQLAuthService.cs`
- `UEModManager/Services/DefaultDatabaseConfig.cs`
- `UEModManager/Views/SupabaseConfigWindow.xaml`
- `UEModManager/Views/SupabaseConfigWindow.xaml.cs`

**修改要点：**
- `AuthenticationService` 完全重写，移除 Supabase SDK，改用 `LocalAuthService`
- `App.xaml.cs` 移除 `SupabaseConfig.ConfigureServices` 和 `SupabaseRestService` DI 注册
- `AuthenticationAgent` 移除 `_postgreSqlAuthService`，login/register 改为纯本地
- `OutputAgent` 的 `DatabaseProvider` 从 `"Supabase"` 改为 `"SQLite"`
- `UEModManager.csproj` 移除 `Supabase` NuGet 包引用

### 2. SharpCompress 0.38.0 → 0.48.1（安全漏洞修复）

**背景：** SharpCompress 0.38.0 存在中危漏洞（NU1902），需升级。但 API 有破坏性变化。

**API 迁移要点：**
```csharp
// 旧 API（0.38.0）
using var archive = ArchiveFactory.Open(stream);
foreach (var entry in archive.Entries) { ... }

// 新 API（0.48.1）
using var reader = ReaderFactory.OpenReader(stream, new ReaderOptions());
while (reader.MoveToNextEntry())
{
    if (reader.Entry.IsDirectory) continue;
    reader.WriteEntryToDirectory(extractPath, new ExtractionOptions
    {
        ExtractFullPath = true, Overwrite = true
    });
}
```

**注意：** 需要 `using SharpCompress.Readers` 替代 `using SharpCompress.Archives`。

### 3. 嵌套 ZIP 导入修复（SplitByImportScope）

**问题：** 用户导入一个包含多个内层 ZIP 的外层 ZIP 时，解压到各自的 `*_extracted` 目录后，两个内层 ZIP 中的同名文件（如 `readme.txt`）会被错误合并到同一分组。

**根因：** 旧代码按解压根目录统一分组，不区分来源容器。

**解决：** 在 `ModFileGrouper` 新增 `SplitByImportScope` 方法，按最内层 `*_extracted` 解压根分桶：

```csharp
public static IReadOnlyList<List<string>> SplitByImportScope(
    IEnumerable<string> filePaths, string rootDirectory)
```

每个内层 ZIP 的解压目录独立成为一个 scope，确保不同来源的同名文件不会误合并。

**同时：** `ModManagementService.ProcessNestedArchives` 从递归改为队列式 BFS，避免深层嵌套栈溢出。

### 4. 删除后重启复活（全链路修复）

**问题：** 用户在 UI 删除 MOD → 重启应用 → MOD 重新出现。

**根因：** 旧代码删除链路有多个数据源入口，删除操作只清了一部分：
- `ModManagementService.DeleteModAsync` 只删了旧数据表的 `Mod` 记录
- 没有删 `PackageRepository`（v2.0 新数据源）的索引
- `ObjectStore` 中的包文件没有被清理
- Profile 引用没有移除

**解决：** 统一删除入口为 `MainViewModel.DeletePackageModCoreAsync`：
1. 先删 `PackageRepository` 中的包（含 `allProfiles` 引用检查和 force 参数）
2. 再移除 Profile 中的引用
3. 最后清理旧数据和 `ObjectStore` 文件

**关键教训：** v2.0 引入 PackageRepository + ObjectStore 后，Model 层有两套"真相来源"，任何写操作必须同时更新两处。

### 5. 重命名/预览图重启丢失（持久化源不一致）

**问题：** 用户重命名 MOD → 重启应用 → 名称恢复旧值。换预览图同理。

**根因：** 旧代码只更新了 UI ViewModel 和旧 `Mod` 表，没有更新 `PackageRepository`。v2.0 的 MOD 列表刷新走 `PackageRepository`，看到的是旧数据。

**解决：**
- 重命名：`RenameModAsync` → 统一写 `PackageRepository.DisplayName`
- 预览图：`ChangePreviewAsync` → 统一写 `PackageRepository.UpdatePreviewImageAsync`
- MOD 列表刷新：`RefreshModsAsync` → 改为 `RefreshFromRepositoryAsync`，以 PackageRepository 为唯一真相源

**关键教训：** 持久化链路中，必须确定 **单一真相源**。v2.0 中 PackageRepository 是 MOD 元数据的真相源，旧 Mod 表仅作兼容读取。

### 6. 启用/禁用不生效（部署与元数据分离）

**问题：** 用户点启用 → UI 显示已启用 → 重启后发现实际没启用。

**根因：** 旧代码有两层 bug：
1. `DeployToggleAsync` 在部署失败或 no-op 时不调 `SetPackageEnabledFlagAsync`，导致元数据没写
2. `SetPackageEnabledFlagAsync` 在 `entry.IsEnabled == enabled` 时直接 return，即使之前写失败也不会补写

**解决：**
- `DeployToggleAsync` 在部署完成（含 no-op 路径）后都调用 `SetPackageEnabledFlagAsync`
- `SetPackageEnabledFlagAsync` 增加早期返回检查但确保部署成功后才写元数据
- 方法重命名为 `SetPackageEnabledFlagAsync`（明确"仅元数据"语义），旧名标 `[Obsolete]`

### 7. 插件/配置文件无法指定目标目录（PluginTargetPath → TargetRootPath）

**问题：** 旧代码仅支持 MOD 文件导入（统一放到 `modPath/PackageKey/`），插件 DLL 和配置文件（`.ini`、`.cfg`）必须放到游戏的特定子目录才能生效（如 `Binaries/Win64/`、`Saved/Config/WindowsNoEditor/`），但导入流程没有提供选择目标目录的入口。

**根因：** `Package` 模型只有 `PluginTargetPath` 字段，且仅在 Plugin 类型使用，Config 类型完全没有目标路径概念。

**解决——全链路改动：**

| 层 | 改动 |
|----|------|
| **模型层** | `Package.TargetRootPath` 新增（Plugin + Config 通用），`PluginTargetPath` 改为 get/set 映射到 `TargetRootPath` |
| **Profile 层** | `ProfilePackageEntry.TargetRootPath` 新增，允许用户按 Profile 覆盖目标路径 |
| **导入确认 UI** | `ImportConfirmDialog` 新增 `TargetPathPanel`：TextBox + ComboBox 快捷路径（Binaries/Win64、Content/Paks/~mods、Saved/Config/WindowsNoEditor），仅非 MOD 包显示 |
| **导入服务** | `ImportAsync` / `ImportSingleAsync` / `ImportDirectFileAsync` / `ImportCompressedAsync` / `ImportFilesAsPackageAsync` / `ImportAsPluginAsync` 全部新增 `targetRootPath` 参数 |
| **部署路径** | `DeploymentTargetPathBuilder.ComputeTargetPath` 从仅 Plugin 改为非 MOD 通用：`gamePath/{TargetRootPath}/{PackageKey}/{file}` |
| **冲突检测** | `ConflictDetector` 同步使用 `TargetRootPath` |
| **部署扫描** | `DeploymentPlanner.ScanDeployedFiles` 中插件扫描改为非 MOD 通用扫描 |

**路径公式：**
- MOD → `modPath/PackageKey/file`
- 非 MOD（Plugin/Config）→ `gamePath/TargetRootPath/PackageKey/file`

### 8. 全选后批量操作

**问题：** 界面有全选 checkbox，但全选后没有批量操作按钮。

**解决：**
- `ModListViewModel` 新增 `SelectedCount` / `HasSelectedMods`、`EnableSelectedAsync` / `DisableSelectedAsync` / `ToggleSelectedAsync`
- `MainWindow.xaml` 新增 `BatchActionsPanel`，绑定 `HasSelectedMods` / `SelectedCount`，含启用/禁用/卸载按钮
- `MainViewModel` 新增 `ToggleModsAsync` / `DeletePackageModsAsync` 批量方法，N 个操作只刷新一次 UI
- 删除顺序改为先仓库后 Profile（避免仓库删除失败但 Profile 引用已丢失）

### 9. 其他修复

- **详情弹窗换预览图不即时显示：** `UpdatePreviewImage` 从 `PreviewImagePath` 重新加载 `BitmapImage`（`IgnoreImageCache`）
- **RAR/7z 导入被阻止：** `ImportDialog.ProcessFiles` 和 `MainWindow.OpenImportWizardAsync` 两处检测并弹窗引导手动解压
- **ENABLE_ALL/DISABLE_ALL/DELETE_SELECTED 多次全量刷新：** 改为批量方法，操作后只刷新一次

## 踩坑记录：核心 Bug 与解决思路

### 坑 #1：多数据源导致写入不生效

**症状：** 改名、换预览图、删除操作在 UI 显示成功，重启后恢复原状。

**诊断方法：** 跟踪数据流——UI → ViewModel → Service → 哪个存储？发现 v2.0 有两套数据源（旧 Mod 表 + 新 PackageRepository），写操作只写了一边。

**解决原则：** 确定 PackageRepository 为唯一真相源，所有写操作统一走它。旧 Mod 表降级为只读兼容。

**适用场景：** 任何"操作后刷新就没了"的 bug，第一反应就应该是"是不是写错了数据源"。

### 坑 #2：嵌套 ZIP 文件冲突

**症状：** 两个内层 ZIP 各有一个 `readme.txt`，导入后变成了同一个 MOD 的文件。

**诊断方法：** 打日志看解压路径和分组逻辑——发现两个 ZIP 解压到各自的 `_extracted` 子目录，但分组时按全局文件列表统一处理。

**解决：** `SplitByImportScope` 按解压根分桶，"哪个 ZIP 出来的就归哪个 scope"。然后每个 scope 独立走文件分组逻辑。

### 坑 #3：SharpCompress 大版本 API 迁移

**症状：** 升级 0.38.0 → 0.48.1 后编译失败，`ArchiveFactory.Open` 不存在。

**教训：** 这种大跨度升级（跨 10 个小版本）一定有 breaking changes。步骤：
1. `dotnet build` 看编译错误
2. 查 NuGet 包的新 API 文档或用反射查看新 API
3. Reader-based API 替代 Archive-based API
4. `using SharpCompress.Readers` 替代 `using SharpCompress.Archives`

### 坑 #4：启用/禁用的"两部曲"

**症状：** 启用操作后 UI 显示已启用，实际文件没有部署。

**根因：** 启用 = 部署文件（DeploymentPlanner）+ 写元数据（ProfileService）。旧代码在这两步之间没有保证原子性——部署失败时不写元数据是对的，但部署成功时也可能漏写元数据，或者 no-op 时直接跳过元数据写入。

**教训：** 有副作用的操作如果涉及多个数据源，画一张状态流转图，确保每个成功路径都覆盖了所有数据源的写入。

### 坑 #5：字段演进的范围扩展

**症状：** `PluginTargetPath` 只能用于 Plugin，Config 类型也需要但没地方存。

**解决思路：**
1. 先在模型层加 `TargetRootPath`（不加 get/set 逻辑，只加字段）
2. 让 `PluginTargetPath` 成为 `TargetRootPath` 的别名（兼容旧代码）
3. 逐层替换引用：Core 纯函数 → 主项目 Service → UI
4. 最后加测试验证

**教训：** 字段重命名/扩展时，先加新字段再让旧字段成为别名，比直接全局替换安全得多。别名保证编译通过，然后可以逐个调用点迁移。

### 坑 #6：删除顺序的依赖风险

**症状：** 删除 MOD 时先移除 Profile 引用再删仓库。如果仓库删除失败，Profile 引用已经没了，MOD 变成"幽灵"——在仓库里但不在任何 Profile 中。

**解决：** 先删仓库（带 `allProfiles` 检查），成功后再移除 Profile 引用。失败则整个操作回滚。

**教训：** 有依赖关系的删除操作，先删被依赖方，再删依赖方。如果被依赖方删除失败，依赖方保持原样。

## 核心运行流程

### 导入包（v2.0.2 版）

1. 用户拖拽文件到 ImportDialog
2. 检测 RAR/7z → 弹窗引导手动解压
3. `ProcessNestedArchives` 队列式 BFS 解压所有嵌套 ZIP
4. `SplitByImportScope` 按最内层 `*_extracted` 分桶
5. 每个 scope 独立 `GroupModFilesByPrefix` 分组
6. 弹出 `ImportConfirmDialog`——非 MOD 包显示 TargetPathPanel 让用户选目标目录
7. 确认后 `ImportAsync(toImport, targetRootPath)` 全链路传递路径
8. `PackageKindDetector` + `ArtifactTypeDetector` 识别类型
9. 写入 `ObjectStore`，更新 `PackageRepository`

### 部署（启用/禁用）

1. UI 调 `MainViewModel.DeployToggleAsync(packageKey, enable)`
2. `DeploymentPlanner.CreateTogglePlanAsync` → `TogglePlanBuilder.BuildToggleOperations`
3. `DeploymentService.ExecuteAsync(plan)` —— 创建事务 → 备份 → 执行 → 提交
4. 成功后 `ProfileService.SetPackageEnabledFlagAsync` 同步元数据

### 路径计算

```
MOD:     modPath / PackageKey / RelativeTargetPath
非MOD:   gamePath / TargetRootPath / PackageKey / RelativeTargetPath
```

## 绝对不要做的事

- ❌ 在 `UEModManager.Core` 里加 `using System.Windows.*` —— TFM 是 `net8.0`，不是 `net8.0-windows`
- ❌ 在 Core 里做 IO（`File.*` / `Directory.*` / `HttpClient`）—— 用 Func 注入或放主项目
- ❌ 在写操作中只更新一个数据源 —— 检查是否需要同步更新 PackageRepository + ObjectStore + 旧表
- ❌ 在事务执行中静默吞异常 —— 累积到 `RollbackFailures`
- ❌ 提交 `bin/` `obj/` `*.log` `.playwright-mcp/` `.claude/`
- ❌ 删除操作先删引用再删实体 —— 先删实体，成功后再清引用
