# 1. 30 秒快速理解

**现在是什么：** 一个 .NET 8.0 多项目解决方案，管理虚幻引擎游戏 MOD，支持 9 款游戏。
解决方案含 **4 个项目**：

- `UEModManager/` — WPF 主程序（IO / UI / Service 编排）
- `UEModManager.Core/` — 纯 Domain 内核（net8.0，单依赖 Newtonsoft.Json）
- `UEModManager.Core.Tests/` — xUnit 单元测试（**552 个测试**）
- `samples/UEModManager.SampleAdapter/` — 第三方扩展 SDK 示例

**要变成什么（已达成）：** 以 **Instance/Profile** 为中心、以 **Package Repository** 为存储、
以 **ResolvedView** 为核心、以 **事务部署** 为落地手段的**多引擎游戏扩展编排平台**。

**用户看到的变化：**
- 同一游戏可保存多套"方案"（MOD 组合），一键切换
- MOD/插件/配置三种类型一目了然，各自有开关
- 冲突不再是黑箱，用户看得见"谁赢了、为什么"（**Phase 4 已修复，2026-04-28**）
- 部署前能预览"哪些文件会改"，一键确认
- 启动前自动检查（**崩溃恢复 + 健康检查，2026-04-28 加入**），绿灯亮了再启动
- **方案可分享、可复现**：lock JSON 轻量分享 + bundle ZIP 完整整合包（2026-04-28 加入）
- **诊断包导出**：报 bug 时一键打包脱敏的日志/数据/事务（2026-04-28 加入）

**你的起点：** Phase 0-13 + Phase 4 修复 + Core 真扎实化全部完成。除 Phase 10 VFS（研究性质）外的全部 v2.0 工作均已落地。**下一步是手动 Windows 验收，然后打 v2.0-rc1 标签发布。**

# 3. v2.0 目标与核心概念

## 3.1 核心概念对照

| v1.8.0 概念 | v2.0 概念 | 用户名称 | 说明 |
|------------|----------|---------|------|
| 游戏 (GameType) | **HostDefinition** | 游戏 | 宿主定义，含引擎类型、路径、适配器 |
| MOD 列表 (ModInfo) | **Package** | MOD / 插件 / 配置 | 统一包模型，含 Artifacts |
| *(不存在)* | **InstanceProfile** | 游戏方案 | 一个游戏可有多套方案 |
| 启用/禁用 (直写目录) | **DeploymentTransaction** | 应用变更 | 事务化部署，可回滚 |
| 冲突检测 (CUE4Parse) | **ConflictRecord** | 冲突 (胜者/败者) | 完整冲突链追踪 |
| *(不存在)* | **ResolvedView** | 生效预览 | 最终视图 = 所有层叠加的结果 |

## 3.2 对用户来说变了什么

**核心交互不变：** 看到 MOD → 开关 → 生效。

**新增能力：**
1. **游戏方案** — 同一游戏多套 MOD 组合，一键切换
2. **类型分类** — MOD(.pak) / 插件(.dll) / 配置(.ini) 一目了然
3. **冲突可见** — 胜者/败者链，用户可手动覆盖
4. **部署预览** — 改之前看清楚会动哪些文件
5. **启动预检** — 绿灯亮了再走

# 10.7 Phase 7：Config Merge ✅ 已完成

> 完成日期：2026-03-27
> 让 ini/cfg/json 配置层从普通文件替换升级为键级合并

## 实际实现

### ① ConfigModels → `Models/ConfigModels.cs`

完整配置合并模型体系：
- **ConfigFormat** 枚举：Ini/Json/Yaml/Toml/Cfg/Unknown
- **ConfigMergeStrategy** 枚举：ReplaceFile/MergeByKey/Append/Patch
- **ConfigEntry** — 键值对（Section + Key + Value）
- **ConfigPatchOperation** — 单个补丁操作（Add/Update/Remove + 来源追踪）
- **ConfigMergePlan** — 合并计划（基础内容 + 来源列表 + 策略 + 冲突）
- **ConfigKeyConflict** — 键级冲突（胜者/败者链）
- **ConfigMergeResult** — 合并结果（内容 + 来源追踪 + 冲突 + 错误）
- **ConfigEntrySource** — 每个键的来源追踪

### ② IConfigParser + 三种解析器

- **IConfigParser** → `Services/Config/IConfigParser.cs` — 解析器接口
- **IniParser** → `Services/Config/IniParser.cs` — INI 格式（[Section]/Key=Value/注释）
- **JsonConfigParser** → `Services/Config/JsonConfigParser.cs` — JSON 格式（嵌套→展平路径）
- **CfgParser** → `Services/Config/CfgParser.cs` — CFG 格式（纯 Key=Value）

### ③ ConfigMergeEngine → `Services/Config/ConfigMergeEngine.cs`

核心合并引擎，4 种策略：
- **MergeByKey** — 按 Section/Key 合并，高优先级覆盖低优先级，自动检测键级冲突
- **Patch** — 仅修改指定键，保留基础配置其余内容
- **Append** — 所有来源追加到基础之后
- **ReplaceFile** — 整文件替换（最高优先级包胜出）

功能特性：
- 自动格式检测（按扩展名 + 内容探测）
- 键级冲突检测（多包修改同一键）
- 来源追踪（每个键的最终来源包）
- 预览模式（`PreviewAsync` 不写文件）

# 10.8 Phase 8：ResolvedView 最终视图 ✅ 已完成

> 完成日期：2026-03-27
> 正式引入"最终运行视图"作为系统核心对象

## 实际实现

### ① ResolvedView + ResolvedEntry → `Models/ResolvedView.cs`

- **ResolvedEntry** — 单条生效文件（目标路径 + 源路径 + 来源层 + 包信息 + 冲突标记）
- **ResolvedEntrySource** 枚举 — HostBase / Package / ConfigMerge / Generated / UserOverride
- **ResolvedView** — 完整视图（条目列表 + 冲突 + 配置合并结果 + ViewHash）
  - `ComputeViewHash()` — SHA-256 稳定哈希（相同输入 = 相同哈希）
  - `IsIdenticalTo()` — 快速比较两个视图是否相同
  - 统计属性：TotalEntries / PackageEntries / ConfigMergeEntries / ConflictCount / TotalSize

### ② ResolvedViewBuilder → `Services/ResolvedViewBuilder.cs`

4 层构建流程：
1. **Package 文件层** — 收集所有已启用包的 Artifact，按优先级排列
2. **冲突解决** — 多包争同一路径时，优先级最高的胜出，生成 ConflictRecord
3. **配置合并层** — 多包修改同一配置文件时，调用 ConfigMergeEngine 键级合并
4. **生成物层** — 纳入活跃的 UserFix 生成物（最高优先级覆盖）

API：
- `BuildAsync()` — 为当前活跃 Profile 构建视图
- `BuildForProfileAsync(profile)` — 为指定 Profile 构建
- `IsViewStaleAsync(view)` — 检查视图是否过期

## 16.0 接手时的发现

进场审查发现 4 个关键问题（按严重度倒序）：

1. **`UEModManager.Core` 项目是空壳**：仅含 `Class1.cs`（VS 默认空模板）和 `DummyClasses.cs`（无引用的旧 Mod 类，26 KB 死代码）。主项目挂着 `<ProjectReference>` 但全部源码引用都在 `obj/` 构建产物里——也就是说 Core 项目自 Phase 1 起从未真正承担 Domain 角色，所有 v2.0 代码都堆在 WPF 主项目里。
2. **测试体系完全不存在**：解决方案中 0 个 `*.Tests.csproj`，`UEModManager.Tests` 只是 `App.xaml.cs` 里的 DEBUG 内联调用。Phase 11 工程硬化在文档上声称完成，实际未起步。
3. **结构化日志缺失**：主项目 153+ 处 `Console.WriteLine`，无时间戳、无级别、无 category，诊断时无法 grep 过滤。
4. **Phase 4 文件路径冲突检测设计漏洞**：`ConflictAnalyzer.ComputeTargetPath` 公式 `modPath/{PackageKey}/{file}` 让两个不同包永远不会落在同一目标路径——意味着 `ConflictType.Path` 这一层的检测**永远不会触发**。Phase 4 验收点"用户能看见胜者/败者链"在路径冲突维度上从未真正成立。

针对这些发现，本轮按"质量为先 + 不着急"原则全部修复。

## 16.5 Phase 4 设计漏洞修复（findings 文档）

### 问题

文件路径冲突检测在当前部署模型下永不触发。

`ComputeTargetPath` 公式：
- Mod: `modPath/{PackageKey}/{RelativeTargetPath}`
- Plugin: `gamePath/{PluginTargetPath}/{PackageKey}/{RelativeTargetPath}`

每个包都自带一层 `{PackageKey}` 子目录。两个不同 PackageKey 的包永远不会落在同一目标路径，
所以 `ConflictAnalyzer` 检测的 dict key（绝对路径）永远只有 1 个 owner，
`if (owners.Count <= 1) continue;` 直接跳过。

这与 `DeploymentPlanner.ComputeTargetPath` 用同一公式——所以**不是实现 bug，是部署模型决定的语义空洞**。

### 修复（方案 B）

新增 `ConflictDetector.ComputeLoadConflictKey`：归一化"无 PackageKey 子目录"的路径作为冲突 key。

- Mod: `modPath/{RelativeTargetPath}`（不含 PackageKey）
- Plugin: `gamePath/{PluginTargetPath}/{RelativeTargetPath}`（不含 PackageKey）

多个包声明相同 `RelativeTargetPath` → 落入同一 key → 视为**加载顺序冲突候选**。
`ConflictRecord.Type` 由 `Path` 改为 **`LoadOrder`**（语义更准）。

### 不变的部分

- `DeploymentPlanner` / `DeploymentService` 仍用旧 `ComputeTargetPath` 实际部署
- 备份/回滚路径模型不变
- 真正的运行时同名覆盖（UE 引擎按加载顺序决定胜者）由引擎处理，不在静态检测范围

### 文档化

完整修复细节：[`docs/findings/2026-04-28-conflict-detector-noop-by-design.md`](docs/findings/2026-04-28-conflict-detector-noop-by-design.md)。
