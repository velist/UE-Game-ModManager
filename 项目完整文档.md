# 爱酱 MOD 管理器项目文档

## 1. 项目背景

### 1.1 项目定位

爱酱 MOD 管理器是一款面向普通玩家的游戏 MOD 管理工具，目标是降低玩家安装、启用、禁用、整理和恢复 MOD 的门槛。

传统 MOD 使用方式通常依赖手动复制文件、覆盖目录、记忆游戏路径和 MOD 格式。普通用户经常会遇到以下问题：

- 不知道 MOD 应该放到哪个目录；
- 启用多个 MOD 后忘记哪些文件被改过；
- MOD 冲突时无法判断哪个生效；
- 删除 MOD 后游戏仍然残留文件；
- 游戏更新或 MOD 安装失败后难以恢复；
- 换电脑或重装系统时配置难以迁移。

本项目的核心目标不是做一个面向技术玩家的文件工具，而是做一个普通玩家也能直接理解和使用的 MOD 管理器。

一句话定位：

> 把 MOD 拖进窗口，拨一下开关，就能安全启用、禁用和恢复。

## 2. 产品定位与边界

### 2.1 当前产品定位

爱酱 MOD 管理器当前主要服务于 PC 游戏 MOD 管理场景，重点解决本地 MOD 管理问题。

核心价值包括：

- 自动识别游戏与 MOD 目录；
- 支持常见 MOD 包格式；
- MOD 导入、分类、启用、禁用、删除；
- 自动备份原文件；
- 安全回滚与失败恢复；
- 冲突检测与部署预览；
- 本地数据透明存储；
- 支持更多游戏与更多引擎扩展。

### 2.2 不是 UE-only 工具

早期项目名称和实现中带有较强的 Unreal Engine 痕迹，但产品定位已经调整：

> 管理器不只局限于 UE 引擎游戏。

当前可以表达为：

- 首发适配部分 UE 游戏；
- 正在扩展更多引擎与更多游戏；
- 产品定位是通用游戏 MOD 管理器，而不是“专为 UE 打造”的工具。

在官网、幻灯片、介绍文案和用户宣传中，应避免使用以下表述：

- 专为 UE 游戏打造；
- 只支持 Unreal Engine；
- UE MOD 专用管理器；
- UE-only。

更合适的表达是：

- 支持主流游戏 MOD 管理；
- 已适配多款热门作品；
- 首发支持部分 UE 大作，并持续扩展更多游戏；
- 面向普通玩家的通用 MOD 管理器。

### 2.3 云端能力不是核心卖点

项目中存在云端认证、邮箱验证码、账号登录等能力，但云端同步并不是当前产品的主要优势。

原因：

- 普通玩家异地登录 MOD 管理器的需求很低；
- MOD 文件本身体积大，不适合通过账号云同步；
- 当前同步更多是配置级同步，不等于插件文件同步；
- 如果宣传“云端同步”，容易造成用户误解。

因此，对外介绍应弱化云端能力，重点突出：

- 本地管理；
- 本地备份；
- 本地恢复；
- 本地配置透明；
- MOD 启停；
- 分类整理；
- 冲突检测；
- 安全回滚。

## 5. 核心功能现状

### 5.1 游戏识别

当前管理器支持游戏实例配置和引擎识别。

主要能力：

- 自动识别已支持游戏；
- 维护游戏安装目录；
- 根据游戏或引擎类型生成部署目标路径；
- 支持用户自定义游戏路径；
- 为后续多引擎扩展保留模型空间。

相关核心模块：

- `EngineDetector`
- `EngineType`
- `InstanceProfile`
- `GameConfigService`
- `DeploymentTargetPathBuilder`

### 5.2 MOD 导入

MOD 导入是当前产品的核心流程之一。

当前能力包括：

- 拖拽导入；
- 支持压缩包；
- 支持常见 MOD 文件；
- 自动分组；
- 自动识别 MOD 文件结构；
- 支持“套娃压缩包”拆解；
- 导入后进入本地仓库管理。

相关模块：

- `PackageImportService`
- `ModFileGrouper`
- `Package`
- `ObjectStore`

### 5.3 MOD 启用与禁用

管理器通过部署规划实现 MOD 启用和禁用，而不是简单复制文件。

核心能力：

- 启用前生成部署计划；
- 部署前备份原文件；
- 禁用时撤回 MOD 文件；
- 恢复原始文件；
- 支持失败检测；
- 支持中断后的恢复流程。

相关模块：

- `DeploymentPlanner`
- `DeploymentTargetPathBuilder`
- `ModManagementService`
- `BackgroundManager`

### 5.4 分类与排序

当前 MOD 支持分类和类型管理。

主要能力：

- MOD 分类；
- 左侧分类栏；
- 右键菜单移动分类；
- 拖拽到分类；
- 多选批量操作；
- 优先级排序。

当前分类相关结构包括：

- `Mod.Categories`
- `Mod.Type`
- `CategoryList_Drop`
- 右键 `ContextMenu`

### 5.5 冲突检测

当前项目已具备冲突检测能力。

目标：

- 检查多个 MOD 是否写入同一目标文件；
- 提示潜在覆盖关系；
- 帮助用户调整加载顺序；
- 避免用户误以为 MOD 没生效。

相关模块：

- `ConflictDetector`
- 部署预览逻辑
- MOD 优先级排序逻辑

### 5.6 本地数据存储

项目强调本地数据透明。

| 数据 | 位置 |
|---|---|
| 用户配置 | `%APPDATA%\UEModManager\` |
| SQLite 数据库 | `%APPDATA%\UEModManager\local.db` |
| MOD 仓库 | `软件目录\Data\Repository\` |
| 原文件备份 | `软件目录\Data\Backups\` |
| 日志 | `软件目录\console.log` |

产品层面应强调：

- 用户数据可见；
- 可手动备份；
- 可复制迁移；
- 卸载后数据可带走；
- 不强依赖云端。

## 6. 认证系统现状

### 6.1 当前认证结构

项目中存在本地认证和云端认证两套能力。

主要模块：

- `AuthenticationService`
- `CloudAuthService`
- `LocalDbContext`
- `CustomOtpService`
- `MailerSendEmailService`
- `AuthenticationAgent`

历史上曾存在或移除过的服务包括：

- `SupabaseOtpService`
- `SupabaseRestService`
- `PostgreSQLAuthService`
- `HybridOtpService`
- `WorkerOtpService`
- `DefaultDatabaseConfig`
- `SupabaseConfig`

当前方向是简化认证体系，避免多套服务并存导致维护复杂度上升。

### 6.2 登录状态问题

历史问题之一是：

> 用户登录成功后，界面仍显示离线模式。

根因：

- `UpdateUserAsync` 只更新数据库；
- 但没有同步更新内存中的认证状态；
- UI 读取的是当前认证状态，而不是单纯读取数据库。

解决方向：

- 登录成功后除了更新数据库，还需要调用类似 `ForceSetAuthStateAsync` 的逻辑；
- 保证 `_currentUser`、本地数据库、UI 状态一致。

### 6.3 UUID 转 int32 问题

云端用户 ID 可能是 UUID，而本地数据库或旧逻辑中使用 int32。

直接转换会导致溢出或格式错误。

解决方案：

- 不直接解析 UUID 为 int；
- 使用稳定哈希；
- 通过 `hash & 0x7FFFFFFF` 保证结果在 int32 正数范围内。

## 8. 已发现的关键技术问题与解决方案

### 8.1 问题一：产品定位过窄

#### 问题表现

早期文案中存在“专为 UE 游戏”“UE 引擎 MOD 管理”等表述。

这会造成两个问题：

1. 用户以为产品只能管理 UE 游戏；
2. 后续扩展其他引擎时，品牌认知会被限制。

#### 解决方案

文案统一调整为：

- 支持主流游戏；
- 首发适配部分 UE 大作；
- 正在扩展更多引擎和游戏。

避免使用：

```text
专为 UE
UE-only
仅支持 UE
```

### 8.2 问题二：云端能力宣传过重

#### 问题表现

早期宣传中把云端账号、云端同步作为卖点之一。

但真实用户场景中：

- 玩家很少异地登录 MOD 管理器；
- MOD 文件不会随账号同步；
- 配置同步价值有限；
- 宣传云端同步容易过度承诺。

#### 解决方案

云端能力保留为辅助功能，但不作为主卖点。

产品宣传重点改为：

- 本地安全；
- 本地管理；
- 一键启停；
- 自动备份；
- 失败恢复；
- 冲突检测；
- 数据透明。

### 8.3 问题三：登录后仍显示离线模式

#### 问题表现

用户完成登录后，界面仍显示“离线模式”。

#### 根因

数据库状态和内存认证状态没有同步。

典型情况：

```text
UpdateUserAsync 已更新数据库
但 _currentUser 没有更新
UI 仍读取旧状态
```

#### 解决方案

登录成功后必须同步更新认证状态。

处理原则：

1. 云端登录成功；
2. 同步用户到本地；
3. 更新本地数据库；
4. 强制设置当前认证状态；
5. 通知 UI 刷新。

### 8.4 问题四：UUID 转 int32 溢出

#### 问题表现

Supabase 或云端认证返回 UUID，而本地旧模型需要 int32 用户 ID。

直接转换会失败。

#### 解决方案

采用稳定哈希映射：

```csharp
var userId = hash & 0x7FFFFFFF;
```

原则：

- 不直接 parse UUID；
- 不依赖随机数；
- 同一个 UUID 应稳定映射到同一个本地 ID；
- 避免负数；
- 避免 int32 溢出。

### 8.5 问题五：密码重置邮件发送失败

#### 问题表现

邮件服务 API Key 在环境变量中存在换行符或不可见字符，导致请求失败。

#### 根因

部署平台保存 secret 时可能带入 `\r\n`。

#### 解决方案

读取 API Key 时进行清理：

```ts
apiKey.trim().replace(/[\r\n]/g, '')
```

原则：

- 所有环境变量进入外部 API 前应 trim；
- secret 不应打印到日志；
- 错误日志只记录状态，不记录敏感内容。

### 8.6 问题六：JSON 字段大小写不一致

#### 问题表现

前后端字段命名不一致。

例如：

```json
access_token
```

C# 模型中可能写成：

```csharp
AccessToken
```

如果没有显式映射，会导致反序列化失败。

#### 解决方案

统一 API 使用 snake_case。

C# 模型使用：

```csharp
[JsonPropertyName("access_token")]
public string AccessToken { get; set; }
```

原则：

- API 字段稳定；
- 客户端通过 attribute 映射；
- 不依赖默认大小写转换行为。

### 8.7 问题七：部署失败后的数据一致性

#### 问题表现

MOD 启用、禁用过程中，如果程序崩溃或文件操作失败，可能出现：

- 数据库显示已启用；
- 实际文件未部署完整；
- 原文件已备份但未恢复；
- 用户不知道当前状态。

#### 解决方案

引入更明确的部署状态和恢复机制。

处理原则：

1. 部署前生成计划；
2. 部署前写入待处理状态；
3. 操作文件；
4. 成功后提交状态；
5. 失败时记录失败状态；
6. 下次启动检测未完成任务；
7. 引导用户恢复或回滚。

### 8.8 问题八：MOD 导入结构复杂

#### 问题表现

MOD 包来源复杂，可能出现：

- 单个 pak 文件；
- zip / rar / 7z 压缩包；
- 压缩包里还有压缩包；
- 多层目录；
- 文件放在错误层级；
- 同包内多个 MOD；
- 配置文件和资源文件混合。

#### 解决方案

通过 `ModFileGrouper` 和导入服务进行结构分析。

处理方向：

- 自动展开压缩包；
- 识别常见 MOD 文件；
- 合理分组；
- 保留原始文件；
- 生成导入结果；
- 给用户明确反馈。

### 8.9 问题九：测试需要覆盖全链路

#### 问题表现

单点修复容易造成其他流程回归。

例如只修导入，不验证部署；只修部署，不验证回滚；只修 UI，不验证数据状态。

#### 解决方案

核心问题必须全链路检查。

重点链路：

```text
导入 MOD
→ 识别文件
→ 生成包
→ 写入数据库
→ 生成部署计划
→ 启用
→ 备份
→ 禁用
→ 恢复
→ 删除
→ 清理残留
```

涉及模块：

- Core 模型；
- 导入服务；
- 部署规划；
- 冲突检测；
- 本地持久化；
- UI 状态刷新；
- 测试用例。

## 13. 后端部署说明

Cloudflare Workers 后端主要用于认证、邮件、密码重置等能力。

常见接口：

```text
POST /api/auth/login
POST /auth/reset
GET  /reset-password
```

部署步骤：

```bash
cd cf-workers/modmanger-api
npm install
wrangler secret put SUPABASE_URL
wrangler secret put SUPABASE_ANON_KEY
wrangler secret put BREVO_API_KEY
npm run deploy
```

注意：

- secret 不得提交到仓库；
- API Key 读取后需要清理换行；
- 日志中不得输出 token、验证码、密钥；
- 认证失败应返回安全的错误信息。

---

## 14. 安全原则

### 14.1 本地密码安全

- 本地密码使用 BCrypt 哈希；
- 不保存明文密码；
- 不在日志中打印密码；
- 不把验证码写入普通日志。

### 14.2 云端密钥安全

- Cloudflare / Supabase / 邮件服务密钥使用环境变量；
- 不提交 `.env`；
- 不输出密钥；
- 不在前端硬编码服务端密钥。

### 14.3 文件操作安全

MOD 管理器涉及大量本地文件操作，应遵循：

- 启用前备份；
- 删除前确认状态；
- 禁用时恢复原始文件；
- 不跨越目标游戏目录乱写文件；
- 失败后保留恢复信息；
- 避免误删用户文件。

### 14.4 用户隐私

诊断包应脱敏：

- 不包含密码；
- 不包含 token；
- 不包含验证码；
- 不包含完整敏感路径；
- 不包含用户私人文件内容。

## 15. 测试策略

### 15.1 核心测试方向

当前测试重点应覆盖：

- 引擎识别；
- 游戏实例配置；
- MOD 文件分组；
- 包导入；
- 部署目标路径生成；
- 冲突检测；
- 诊断清单生成；
- 数据状态转换；
- 失败恢复。

### 15.2 推荐测试链路

完整链路测试建议覆盖：

```text
导入一个 MOD
→ 分类入库
→ 生成部署计划
→ 启用 MOD
→ 检查备份
→ 禁用 MOD
→ 检查恢复
→ 删除 MOD
→ 检查清理
```

### 15.3 UI 测试原则

WPF UI 当前仍以 code-behind 为主，因此 UI 自动化测试成本较高。

建议：

- 核心逻辑下沉到 Core 或 Service；
- UI 只负责调用和展示；
- 关键逻辑通过单元测试覆盖；
- 重要用户流程通过人工验收或端到端验证覆盖。

## 16. 当前主要风险

### 16.1 code-behind 较重

当前 UI 是旧式单体 code-behind 结构。

风险：

- MainWindow 逻辑过多；
- UI 与业务耦合；
- 难以测试；
- 后续维护成本较高。

建议：

- 不急于全面 MVVM 重构；
- 新增复杂逻辑优先放入 Service；
- 逐步减少 MainWindow 直接业务判断；
- 避免一次性大重构。

### 16.2 认证链路历史包袱较多

历史上存在多套认证服务：

- Supabase；
- PostgreSQL；
- Workers；
- OTP；
- 本地认证；
- Hybrid 方案。

风险：

- 旧代码残留；
- 状态同步复杂；
- 登录、离线、本地用户之间边界不清晰。

建议：

- 明确当前唯一推荐认证路径；
- 删除无用服务；
- 保持本地优先；
- 云端只做辅助能力；
- UI 不把登录作为核心门槛。

### 16.3 多引擎扩展需要谨慎

产品定位已经不局限 UE，但底层很多模型仍保留 UE 历史痕迹。

风险：

- 文案已扩展，但技术实现还未完全泛化；
- 不同游戏引擎 MOD 部署方式差异大；
- 过早承诺“全引擎支持”会增加风险。

建议：

- 对外表达为“持续扩展更多游戏”；
- 内部逐步抽象部署策略；
- 每个游戏建立明确 profile；
- 不要在没有适配前承诺具体引擎能力。

### 17.3 长期方向

长期可以考虑：

## 18. 项目当前结论

爱酱 MOD 管理器已经从早期“能用的 UE MOD 管理工具”逐步升级为“面向普通玩家的通用 MOD 管理器”。

当前最重要的产品方向是：

- 不再把 UE 作为唯一定位；
- 不把云端同步作为核心卖点；
- 坚持本地、安全、透明、可恢复；
- 把复杂技术能力包装成普通用户能理解的操作；
- 把导入、部署、备份、回滚这些核心链路做稳；
- 对外展示时少讲技术，多讲用户收益。

最终目标是：

> 让玩家不用懂路径、不用懂格式、不用怕改坏文件，也能安心使用 MOD。
